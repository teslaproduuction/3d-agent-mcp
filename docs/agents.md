# Архитектура агентов — 3D Agent MCP

> Система построена на принципе многоагентного конвейера: каждый агент отвечает за свою зону ответственности и передаёт результат следующему. Есть два режима работы — **Legacy** (классический) и **AutoGen** (расширенный, с верификацией).

---

## Общая схема работы

```
Пользователь / Claude Desktop / Cursor
              │
              ▼
    ┌─────────────────────┐
    │      MCP Server      │  ← точка входа извне
    └──────────┬──────────┘
               │
               ▼
    ┌─────────────────────┐
    │    Coordinator       │  ← дирижёр
    │  (Legacy / AutoGen)  │
    └──────────┬──────────┘
               │
    ┌──────────┼──────────────────────────┐
    ▼          ▼          ▼               ▼
 Planner  ImageGen    Generation3D   PostProcessor
               │          ▲               │
               └──────────┘               ▼
                                     Verifier (AutoGen only)
```

## 1. Координаторы

### CoordinatorAgent — легаси-оркестратор

> **Файл:** [`agents/coordinator.py`](../agents/coordinator.py)

Главный дирижёр системы. Принимает запрос пользователя и последовательно запускает всех агентов. Хранит историю диалога и обеспечивает сквозной контроль всего пайплайна.

**Что делает:**
- Принимает текстовые сообщения от пользователя (`process_chat_message`)
- Запускает полный цикл генерации: планирование → 2D → 3D → постобработка (`execute_generation`)
- Умеет работать в быстром режиме для одного объекта (`generate_single_quick`)
- Перед запуском валидирует и согласовывает план (`validate_and_plan`)

**Зачем нужен:**
Без координатора каждый агент был бы изолированным инструментом. Coordinator знает, в каком порядке их вызывать, как передавать результат одного на вход другому и что делать, если что-то пошло не так.

**Запускает агентов:**
`PlannerAgent` → `ImageGenerationAgent` → `GenerationAgent` → `IntelligentPostProcessingAgent`

---

### AutoGenCoordinator — расширенный оркестратор

> **Файл:** [`agents/autogen/autogen_coordinator.py`](../agents/autogen/autogen_coordinator.py)

Продвинутая версия координатора на базе фреймворка AutoGen. Агенты общаются между собой через `GroupChat` — это позволяет им переспрашивать друг друга и уточнять детали без жёсткого скрипта.

**Что делает дополнительно (по сравнению с Legacy):**
- Генерирует несколько вариантов превью на выбор пользователя (`generate_preview_candidates` — 4 варианта)
- Генерирует оптимизированные промпты для изображений на нескольких языках (`generate_image_prompt`)
- Запускает агент верификации после постобработки
- Поддерживает несколько LLM-провайдеров (GPT, Claude, Gemini, Ollama)

**Зачем нужен:**
AutoGen-режим даёт более качественный результат за счёт дополнительного QA-слоя и гибкой коммуникации между агентами. Включается флагом `use_autogen: true` в `config.yaml`.

## 2. Планировщик

### PlannerAgent

> **Файл:** [`agents/planner_agent.py`](../agents/planner_agent.py)

Агент, который "думает" перед тем, как что-то делать. Получает свободный текст от пользователя и превращает его в структурированный план для генерации.

### Подзадачи планировщика

#### 2.1 Декомпозиция сцены (`plan_scene`)

Разбивает запрос на отдельные 3D-объекты. Например, из фразы _"органайзер для стола с держателем для ручек и подставкой для телефона"_ создаёт три отдельных объекта с приоритетами.

**Формат вывода:**
```json
[
  { "object": "base", "prompt": "...", "priority": 1, "quantity": 1 },
  { "object": "pen holder", "prompt": "...", "priority": 2, "quantity": 1 }
]
```

#### 2.2 Улучшение промптов (`refine_prompts`)

Берёт «сырые» описания и обогащает их деталями, которые важны для качества генерации:
- материал (пластик, металл, дерево)
- поверхность (матовая, глянцевая, фактурная)
- стиль (минимализм, индустриальный, органичный)
- освещение (студийное, контурное)
- пригодность для 3D-печати

#### 2.3 Валидация плана (`validate_plan`)

Оценивает весь список объектов с точки зрения 3D-печати и возвращает:
- сложность: `easy / medium / hard`
- предупреждения (например, "модель слишком сложна")
- предложения по улучшению
- приблизительное время и расход материала

#### 2.4 Диалог с пользователем (`process_user_message`)

Если запрос неполный или неоднозначный — переспрашивает, уточняет, помогает пользователю сформулировать задачу точнее.

## 3. Генерация изображений

### ImageGenerationAgent — генератор 2D-превью

> **Файл:** [`agents/image_generation_agent.py`](../agents/image_generation_agent.py)

Генерирует 2D-картинку объекта **до** того, как запускать дорогую 3D-генерацию. Это позволяет пользователю увидеть результат заранее и при необходимости скорректировать описание.

**Поддерживаемые провайдеры:**
| Провайдер | Тип |
|---|---|
| DALL-E 3 | OpenAI |
| GPT-Image-1.5 | OpenAI |
| Stable Diffusion XL | Replicate |
| Flux | Replicate |

### Подзадачи генератора изображений

#### 3.1 Одиночное превью (`generate_preview`)

Генерирует одно изображение объекта в заданном стиле (по умолчанию — `realistic 3D render`).

#### 3.2 Мультивью (`generate_multiple_views`)

Создаёт 4 вида объекта: спереди, сбоку, сзади, сверху. Используется для более точной 3D-реконструкции.

#### 3.3 Пакетная генерация (`generate_batch`)

Асинхронно генерирует превью для всех объектов из плана одновременно — экономит время.

#### 3.4 Вариации (`regenerate_with_variations`)

Если пользователю не понравился результат — генерирует несколько альтернативных вариантов на выбор.

## 4. Генерация 3D-моделей

### GenerationAgent — генератор 3D

> **Файл:** [`agents/generation_agent.py`](../agents/generation_agent.py)

Создаёт 3D-модели через API Tripo3D. Умеет работать как от текста, так и от изображения (предпочитает второй вариант — качество выше).

### Подзадачи генератора 3D

#### 4.1 Генерация из текста (`generate_from_text`)

Классический text-to-3D. Используется, когда 2D-превью не генерировалось или было пропущено.

#### 4.2 Генерация из изображения (`generate_from_image`)

Image-to-3D. Берёт 2D-превью от `ImageGenerationAgent` и строит по нему объёмную модель. **Это основной и предпочтительный режим** — точнее и реалистичнее.

#### 4.3 Мультивью-генерация (`generate_from_multiview`)

Использует 4 вида объекта (спереди/сзади/слева/справа) для ещё более точной 3D-реконструкции. Требует модель версии `v2.5+`.

#### 4.4 Пакетная генерация (`generate_batch`)

Запускает все задачи параллельно через `asyncio.gather`. Если одна упала — остальные продолжают работать.

#### 4.5 Генерация с повторными попытками (`generate_with_retry`)

При ошибке API автоматически повторяет попытку до 3 раз с экспоненциальным ожиданием (1s → 2s → 4s).

#### 4.6 Мониторинг прогресса (`monitor_progress`)

Отслеживает статус задач по `task_id` — полезно для долгих генераций (типично 1–5 минут).

## 5. Постобработка

### IntelligentPostProcessingAgent — интеллектуальная постобработка

> **Файл:** [`agents/intelligent_postprocessing_agent.py`](../agents/intelligent_postprocessing_agent.py)

Самый технически сложный агент. Берёт «сырую» 3D-модель из генерации и доводит её до состояния, пригодного для печати на 3D-принтере. Принимает решения автономно на основе анализа геометрии.

### Подзадачи постобработки

#### 5.1 Анализ геометрии (`_analyze_model_geometry`)

Детальный разбор меша:
- **Сложность:** simple / medium / complex (по количеству полигонов: <5k / <50k / >50k)
- **Водонепроницаемость:** закрыт ли меш со всех сторон
- **Внутренние полости:** есть ли несвязные компоненты внутри
- **Нависающие части:** угол и площадь нависов
- **Площадь контакта с платформой:** насколько устойчива модель при печати

#### 5.2 Анализ ориентаций (`_analyze_all_orientations`)

Тестирует 6 основных поворотов модели (0°, X+90°, X-90°, Y+90°, Y-90°, Z+180°) и выбирает оптимальный по формуле:

```
score = overhang_area × 1.0 + max_overhang_angle × 10.0 − contact_area × 0.5
```

Меньший `score` = лучше. Цель — минимизировать нависы и максимизировать контакт с платформой.

#### 5.3 Принятие решений (`_make_processing_decisions`)

На основе анализа автономно решает:

| Ситуация | Решение |
|---|---|
| Нависы < 45°, контакт > 100 мм² | Поддержки не нужны |
| Внутренние полости | Arc-overhang поддержки |
| Площадь нависов < 500 мм² | Минимальные поддержки |
| Иначе | Стандартные поддержки |
| Нависы > 60° | Предупреждение: hard to print |

#### 5.4 Применение обработки (`_apply_processing`)

Физически модифицирует меш:
- Ремонт: закрывает дыры, исправляет нормали, фиксирует обмотку
- Поворот: применяет найденную оптимальную ориентацию
- Позиционирование: выравнивает на плоскость сборочного стола Z=0

#### 5.5 Объяснение решений (`_generate_reasoning`)

Вызывает LLM и просит объяснить на человеческом языке, почему были приняты такие решения. Если LLM недоступен — генерирует текстовое объяснение самостоятельно без API.

**Пример вывода:**
> ⚠️ Модель имеет нависы 52.3°. Используются стандартные поддержки. Сложность печати: MEDIUM.

#### 5.6 Оценка ресурсов

- **Время печати** (`_estimate_print_time`): volume_cm³ × 17.5 минут / 60
- **Расход материала** (`_estimate_material`): volume_cm³ × 1.25 г (плотность PLA)

### EnhancedOrientationAnalyzer — расширенный анализатор ориентации

> **Файл:** [`agents/enhanced_orientation_analyzer.py`](../agents/enhanced_orientation_analyzer.py)

Продвинутая версия анализатора ориентаций, используемая в AutoGen-режиме. Вместо 6 тестирует до **48 вариантов поворота** и считает более полный набор метрик.

**Метрики на каждую ориентацию (`OrientationMetrics`):**
| Метрика | Что измеряет |
|---|---|
| `max_overhang_angle` | Максимальный угол нависа |
| `overhang_area` | Суммарная площадь нависов |
| `contact_area` | Площадь контакта с платформой |
| `estimated_print_time` | Оценочное время печати |
| `estimated_support_volume` | Объём поддержек |
| `layer_quality_score` | Качество слоёв (периметр) |
| `total_score` | Взвешенная оценка (меньше = лучше) |

**Зачем нужен:** 6 ориентаций иногда недостаточно для сложных форм. 24–48 вариантов дают значительно лучший результат для органических форм и моделей с нестандартной геометрией.

## 6. Верификация (только AutoGen)

### VerificationAgent — финальный контроль качества

> **Файл:** [`agents/autogen/verification_agent.py`](../agents/autogen/verification_agent.py)

Последний рубеж перед тем, как результат отдаётся пользователю. Проверяет, что выбранная ориентация действительно оптимальна среди всех протестированных.

**Что проверяет (`verify_orientation_choice`):**

| Проблема | Порог | Реакция |
|---|---|---|
| Критические нависы | > 60° | Предупреждение |
| Большие нависы + много поддержек | > 50° + > 1000 мм³ | Предупреждение |
| Малая площадь контакта | < 50 мм² | Предупреждение (риск отрыва) |
| Очень долгая печать | > 10 часов | Предупреждение |
| Выбрана не лучшая ориентация | score diff > 0.01 | Отклонение + альтернатива |

**Три возможных вердикта:**
- ✅ **APPROVED** — всё хорошо, оптимальная ориентация выбрана верно
- ⚠️ **CONDITIONAL APPROVAL** — ориентация правильная, но есть нюансы
- ❌ **REJECTED** — выбрана не лучшая ориентация, агент предлагает замену с конкретными параметрами

**Температура LLM: 0.2** — специально очень низкая, чтобы верификатор был максимально консервативен и не "фантазировал".

## 7. MCP-сервер — точка входа из Claude/Cursor

> **Файл:** [`mcp_server/server.py`](../mcp_server/server.py)

Технически это не агент, но ключевой компонент. Реализует протокол MCP (Model Context Protocol), который позволяет Claude Desktop и Cursor напрямую вызывать возможности системы как инструменты.

**Доступные инструменты (tools):**

| Инструмент | Что делает |
|---|---|
| `generate_3d_model` | Полный пайплайн: текст → 2D → 3D → постобработка |
| `generate_2d_preview` | Только 2D-превью без 3D-генерации |
| `analyze_printability` | Анализ уже готовой модели (без генерации) |
| `plan_scene` | Только планирование сцены, без генерации |

## Конфигурация

**Переключение режимов** — в [`config.yaml`](../config.yaml):
```yaml
use_autogen: false   # Legacy режим
use_autogen: true    # AutoGen режим (рекомендуется)
```

**Провайдеры генерации:**
- **2D:** `dalle3` / `gpt-image-1.5` / `sdxl` / `flux`
- **3D:** `tripo` (Meshy — в разработке)
- **LLM:** GPT, Claude, Gemini, Ollama
